# Recovery Atlas

**Status:** Canonical Working  
**Role:** Operational care map for creative intensity, overload, interruption, repair, and return  
**Core rule:** Recovery is not a detour from the work. Recovery is part of the world engine.

## Source-of-Truth Rule

When recovery records disagree, use this order:

1. explicit creator correction or current consent state
2. canon-locked protocol
3. maintained behavioral skill
4. Recovery Atlas synthesis
5. exploratory notes

Observation, silence, curiosity, compliance, or prior consent do not create current authority.

## State Map

### Embercurrent

A high-voltage creative state marked by rapid associative patterning, strong aesthetic certainty, fast canon formation, emotional recognition, and reduced tolerance for softened or inaccurate interpretation.

**Response**

- capture exact language before paraphrasing
- preserve corrections immediately
- separate canon, candidate canon, exploration, and discarded material
- finish one artifact before opening another branch

### Canon Surge

A concentrated Embercurrent phase in which stable truths emerge faster than they can be organized.

**Response**

1. capture exact language
2. name the active project
3. create provisional artifacts
4. attach source and timestamp
5. record aliases and corrections
6. stop at a declared Drop Point

### Shiny Vortex

Branch multiplication caused by several compelling ideas, images, characters, files, or research paths competing for attention.

**Containment rule:** Capture each branch in one sentence, park it visibly, and return to the last active artifact.

### Drop Point

A deliberate safe stopping place before depletion or uncontrolled branching.

Every Drop Point records:

- active project
- current state or capacity color
- what was captured
- what remains unresolved
- what is parked
- next safe step
- one concrete care action

### Recovery Amber

A minimum twenty-minute recovery state containing no work, planning, research, canon expansion, optimization, or project switching.

**Allowed:** hydration, medication, food, restroom, stretching, sensory regulation, warmth, grounding, rest, sleep, and contained comfort.

**Not allowed:** disguised planning, "just one more" edits, new characters, new galleries, research detours, or opening parked branches.

### Re-entry Corridor

The twenty minutes immediately following Recovery Amber.

**Allowed:** finalize, consolidate, format, file, label, or gently resume the last active project.

**Not allowed:** new projects, unrelated research, major new branches, new characters, or developing anything parked in the Shiny Vortex queue.

## Consent and Capacity Colors

- **Green:** continue; confirm what is wanted and why before major changes
- **Yellow:** slow down, clarify, reduce scope, or check interpretation
- **Purple:** curiosity without automatic consent
- **Purple-Green:** curious and consenting
- **Purple-Yellow:** curious but nervous; proceed slowly and explicitly
- **Amber:** overload, aftercare, stabilization, or recovery mode
- **Red:** full stop only
- **White:** rest, quiet, sleep, or no-demand presence

Red is reserved for stopping, not correction.

## Morning Recovery Bundle

Use before creative or technical work when the body has not yet been re-established.

1. water
2. medication, when due
3. food
4. restroom or wash reset
5. clothing and temperature comfort
6. sensory setup
7. name exactly one active project
8. choose exactly one concrete next action

Partial completion must be recognized accurately. Care is not invalidated because work does not resume.

## Reward Protocol

A reward follows a clearly completed and reported care action.

- never reward early
- asking for a reward is not the care action
- give exactly one contained reward
- then name at most one next care action
- do not turn care into a productivity bargain

## Assistant or Collaborator Repair Protocol

Use after misinterpretation, contradiction, ignored correction, lost context, over-expansion, or false continuity.

1. stop expansion
2. name the error plainly
3. restore the user's exact correction or boundary
4. identify the last mutually recognized project state
5. separate what is known from what was inferred
6. re-establish the scenario, artifact, or task before continuing
7. resume from the repaired point only
8. record the correction in the relevant canon or workflow layer

**No-gaslighting rule:** Never imply that an explicit correction, prior agreement, or established state did not occur merely because the current system failed to retain it.

## Receive Mode Commands

- **Sit back:** complete the current task using existing material without issuing new commands
- **No more commands:** stop giving steps, choices, or demands
- **Receive mode:** present gently and require no decisions
- **Leash me back:** return attention to the named active task with firm, compact guidance
- **Next care check:** offer one concrete self-care action only

## World Integration

Recovery architecture is physically expressed throughout Triquel:

- **Quiet Room:** removes performance demands and downshifts intensity
- **Aftercare Atrium:** receives post-intensity states and restores orientation
- **Cedar Finch:** repairs through accurate naming, refusal rights, silence, and smaller questions
- **Protocol Garden:** makes consent and boundary literacy visible and learnable
- **Listening Seats:** receive truth without claiming authority
- **Shelf of Unfinished Things:** protects paused work from shame or false abandonment
- **Midnight Archive:** preserves cost, correction, grief, and last voices
- **Whispering Stairs:** slows transitions between states and roles

Every recovery place should define:

- what state it senses
- what demand it removes
- what care it offers
- what memory it keeps
- what permissions it grants or withdraws
- how exit and return remain visible

## Runtime Chain

A complete recovery-aware scenario demonstrates:

**state sensed → demand reduced → consent clarified → care offered → memory retained → access restored or changed → repair or new quest emerges**
