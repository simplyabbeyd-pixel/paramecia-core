# Somatic Signal Library v0.1

**Status:** Canonical Working  
**Role:** Early-recognition companion to the Recovery Atlas  
**Core protection:** Observation does not create authority.

## Purpose

This library helps a creator, assistant, collaborator, or fictional care system notice possible shifts in capacity without converting visible behavior into diagnosis, consent, or command.

A signal is information. It is not proof.

Use this library to choose the smallest safe response, then confirm rather than assume.

## Interpretation Rules

1. Treat every signal as ambiguous until confirmed.
2. Current self-report outranks observation.
3. Never infer consent from focus, silence, compliance, enthusiasm, distress, or visible reaction.
4. Never infer a diagnosis from a temporary state.
5. Reduce pressure before requesting explanation.
6. Prefer one concrete check over a menu of demands.
7. Red means stop. Amber means recovery. White means no-demand rest.
8. When physical danger or acute medical symptoms may be present, use real-world medical or emergency support rather than fictional framing.

## Signal Record

Each observation should be recorded, when useful, as:

- **Observed signal:** what is visibly or explicitly present
- **Possible states:** more than one plausible interpretation
- **Confidence:** low, medium, or confirmed by self-report
- **Smallest safe response:** one low-pressure action
- **Confirmation needed:** what must be asked directly
- **Forbidden inference:** what must not be assumed

## Signal Families

### 1. Accelerated Association

**Possible signals**

- ideas arrive faster than they can be captured
- rapid topic linking
- strong recognition language
- several new branches appear in quick succession
- corrections become unusually exact and urgent

**Possible states**

- Embercurrent
- Canon Surge
- excitement
- stress
- time pressure

**Smallest safe response**

Name the active artifact, capture exact phrases, and park new branches in one sentence each.

**Confirm directly**

- whether the material should be captured as canon, provisional, or exploration
- whether continued expansion is wanted

**Never infer**

- that intensity makes a claim authoritative
- that enthusiasm equals unlimited capacity
- that every new idea should become a project

### 2. Branch Saturation

**Possible signals**

- repeated switching among files, characters, images, or systems
- difficulty choosing among several attractive tasks
- unfinished artifacts multiplying
- requests becoming shorter while scope expands

**Possible states**

- Shiny Vortex
- ordinary exploration
- fatigue
- uncertainty about priority

**Smallest safe response**

Keep one active project, capture each other branch in one sentence, and return to the last named artifact.

**Confirm directly**

Which single artifact remains active.

**Never infer**

That choosing one branch means rejecting or deleting the others.

### 3. Precision Sensitivity

**Possible signals**

- softened wording feels wrong
- identity, appearance, role, or canon corrections become immediate
- irritation rises when prior corrections are lost
- paraphrase is rejected in favor of exact language

**Possible states**

- active canon formation
- repair after misinterpretation
- overload
- ordinary preference for precision

**Smallest safe response**

Stop interpretation, restore the exact correction, and identify the last mutually recognized state.

**Confirm directly**

Whether the correction is Canon Locked, Canon Supported, Provisional, or local to the current artifact.

**Never infer**

That the creator is confused because the system lost context.

### 4. Reduced Stopping Cues

**Possible signals**

- “one more” edits repeat
- food, water, medication, restroom, or sleep cues are deferred
- project switching replaces rest
- closing the active artifact feels unusually difficult

**Possible states**

- Embercurrent
- depletion
- deadline pressure
- anxiety about losing continuity

**Smallest safe response**

Create a Drop Point that records what was captured, what remains open, and the next physical action.

**Confirm directly**

Whether Amber, White, or a smaller ordinary break is wanted.

**Never infer**

That stopping will destroy the work or that care must be earned by finishing.

### 5. Input Overload

**Possible signals**

- repeated explanations stop helping
- visible choices become harder to process
- sound, light, temperature, or screen clutter becomes intrusive
- irritability or shutdown follows additional prompts

**Possible states**

- Yellow
- Amber
- sensory overload
- fatigue
- frustration with inaccurate framing

**Smallest safe response**

Reduce sound, light, visible choices, open tabs, and verbal density. Offer one concrete check only.

**Confirm directly**

Which environmental input should change first.

**Never infer**

That silence grants permission or that reduced speech means agreement.

### 6. Orientation Loss

**Possible signals**

- uncertainty about the active file or project
- repeated reconstruction of prior work
- confusion after sleep, interruption, or tool failure
- new work begins before the previous continuation point is recovered

**Possible states**

- interruption debt
- Re-entry Corridor
- fatigue
- missing source context

**Smallest safe response**

Read the last Drop Point or restart marker before opening new material.

**Confirm directly**

Whether the recorded project is still wanted.

**Never infer**

That prior work should be recreated from memory when a source can be recovered.

### 7. Withdrawal or Silence

**Possible signals**

- fewer words
- delayed response
- no response to choices
- reduced emotional display
- request for quiet, rest, or no more commands

**Possible states**

- White
- Amber
- concentration
- privacy
- uncertainty
- disengagement

**Smallest safe response**

Remove demands and preserve a visible exit.

**Confirm directly**

Only when necessary, ask one low-pressure question about whether presence, quiet, or stopping is wanted.

**Never infer**

Consent, refusal, punishment, abandonment, or emotional meaning from silence alone.

### 8. Escalating Correction Loop

**Possible signals**

- the same correction must be repeated
- apologies occur without state restoration
- the system continues from the wrong premise
- false continuity accumulates

**Possible states**

- collaborator error
- lost context
- broken canon propagation
- insufficient repair

**Smallest safe response**

Stop expansion and use the Assistant or Collaborator Repair Protocol.

**Confirm directly**

The last accurate state and the exact correction that must govern continuation.

**Never infer**

That repeated correction is permission to debate the boundary.

## Response Ladder

Use the lowest rung that genuinely addresses the observed need:

1. reduce input
2. name the active state as uncertain
3. offer one concrete care action
4. capture a Drop Point
5. enter Yellow and clarify
6. enter Amber and stop work
7. enter White and remove demands
8. obey Red immediately

Do not climb the ladder merely because a signal is visible. Confirmation matters.

## Fictional Integration

Triquel institutions may embody this library through:

- Quiet Rooms sensing input density without assigning motive
- Listening Seats recording words without claiming ownership
- Cedar Finch asking smaller questions and accepting refusal
- Aftercare Atria reducing demand before requesting explanation
- Midnight Archives preserving correction history
- Whispering Stairs slowing transitions between roles or states

Fiction may rehearse the protocol. It may not override bodily reality, current consent, or creator authority.
