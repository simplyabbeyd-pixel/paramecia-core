# ⛩️ Project Core Integration Map v1.0

> **Purpose:** Provide one operational map connecting Paramecia / Triquel cosmology, canon governance, relational care ecology, characters, regions, scenarios, recovery systems, visual artifacts, and production workflows.

> **Core axiom:** **Nothing flourishes alone.**

> **Core civic rule:** **Observation does not create authority. Care is infrastructure.**

## 1. Authority and Source Hierarchy

The project uses distinct layers of authority. They must not be collapsed into one another.

1. **Versioned canon ledger** — authoritative, reviewable, rollback-capable record.
2. **Notion studio** — readable relational workspace for synthesis, navigation, drafting, and active development.
3. **Project chats and source artifacts** — discovery material, exact phrases, corrections, and recoverable provenance.
4. **Presentations, images, JanitorAI exports, and derived tools** — expressions of canon, never independent sources of authority.

Every major entry should carry:

- canon status
- source or provenance
- correction history
- aliases and deprecated names
- relationship links
- review date or version

## 2. Canon Status Model

Use these statuses consistently across Markdown, JSON, Notion, visual archives, and exports:

- **Canon Locked** — authoritative and stable until formally revised
- **Canon Supported** — consistent with canon but interpretive or incomplete
- **Provisional** — active development material awaiting review
- **Exploration** — generative possibility with no authority
- **Contradiction** — conflicts with a higher-authority source or correction
- **Deprecated** — intentionally retired but preserved for provenance
- **Archive** — historically relevant and inactive

Corrections are first-class canon events. Never silently overwrite a correction trail.

- What consequence follows?

## 5. Consent and Communication Layer

### Shared color states

- **Green:** continue or approach
- **Yellow:** slow down and clarify
- **Purple:** curiosity without consent
- **Red:** stop
- **White:** rest without demand
- **Amber:** regulation and aftercare; stop expansion

Variants may include Purple-Green and Purple-Yellow when curiosity and permission need separate expression.

### Operational principles

- consent is active, legible, ongoing, and withdrawable
- refusal must not erase belonging
- visible exits must remain usable
- witnessing does not create ownership
- role separation reduces ambiguity under intensity
- repair requires changed behavior, not explanation alone

### Cedar Finch protocol

The Repair Lantern protects the right to:

- leave
- pause
- remain silent
- refuse
- answer a smaller question
- correct the frame

## 6. Creative State and Recovery Architecture

The production system must protect the creator as carefully as it protects canon.

### Embercurrent

A high-voltage creative state marked by rapid association, strong aesthetic certainty, fast canon formation, and reduced tolerance for softened interpretation.

### Canon Surge

During a surge:

1. capture exact phrases
2. preserve corrections
3. create provisional artifacts
4. separate canon from exploration
5. stabilize before opening new branches

### Shiny Vortex

Too many attractive branches competing at once. Response: rails, short prompts, one deliverable, one active lane.

### Recovery Amber

Minimum recovery interval with no expansion. Receive mode, grounding, and small contained rewards only after real care actions.

### Re-entry Corridor

Return through consolidation and finishing on the last active project before creating anything new.

## 9. Scenario Runtime

A scenario should produce persistent world consequences rather than evaporating after resolution.

### Runtime chain

**Pressure → Choice → Protocol → Witness → Memory → Access Change → Institutional Response → Ecological or Civic Consequence → Repair Opportunity → Canon Candidate**

Each scenario should define:

- presenting pressure
- affected people and place
- Accord gates
- consent states and visible exits
- what each participant can refuse
- memory written by the encounter
- trust and access changes
- institutional response
- failure state
- repair path
- possible future hooks

## 10. Visual Canon Governance

Visual artifacts should carry one of these statuses:

- Canon Locked
- Canon Candidate
- Variant
- Deprecated
- Reference Only

Every locked image should record:

- represented character or place
- visual corrections
- clothing and cultural markers
- body-form constraints
- source prompt or generation lineage when available
- relationship to written canon

An image may confirm an established detail. It may not independently invent authority.

## 13. Definition of Completion

The integration layer is functioning when:

- every authoritative claim has a source and status
- aliases and deprecated forms are searchable
- corrections cannot disappear during export
- every character, place, institution, and scenario has stable identifiers
- runtime events can update memory, trust, access, place state, and institutional response
- visual artifacts link back to written canon
- recovery protocols govern production pace
- Notion and derived artifacts can be rebuilt from the versioned ledger without losing meaning
