# Knotwork Nexus Integration Architecture

> **Core rule:** Observation does not create authority. Care is infrastructure.

**Status:** Provisional operational architecture  
**Role:** Connect canon, characters, places, scenarios, recovery systems, and artifact production into one governed ecology.

The Knotwork Nexus is the operational center of Paramecia / Triquel. It prevents the project from becoming a stack of isolated lore entries by requiring every major element to participate in relationships, memory, consequences, and repair.

## 1. Six Connected Layers

### World Ecology
Contains cosmology, regions, kin, materials, architecture, institutions, routes, and environmental states.

### Relational Engine
Contains consent, trust, access, memory, role separation, repair, witness, and changing relationships.

### Canon Governance
Contains source records, exact corrections, aliases, status, version history, contradiction tracking, and rollback paths.

### Recovery Architecture
Contains Embercurrent, Drop Points, Recovery Amber, Re-entry Corridor, capacity signals, aftercare, and creator-protection rails.

### Playable Runtime
Contains scenarios, tensions, choices, persistent consequences, NPC memory, place-state change, institutional response, and quest emergence.

### Artifact Pipeline
Contains Markdown, JSON, schemas, diagrams, presentations, image indexes, lorebooks, scenario cards, and release records.

No layer should operate as a sealed jar. A change in one layer should identify which neighboring layers must be reviewed.

## 2. Operational Registers

The Nexus should maintain five primary registers:

1. **Canon Atlas**: approved truths, corrections, source artifacts, aliases, and status
2. **Character Constellation**: people, kin, roles, relationships, tensions, and visual canon
3. **Living Places**: regions, rooms, institutions, sentient architecture, thresholds, and place states
4. **Scenario Loom**: encounters, protocols, hooks, tensions, outcomes, and persistent changes
5. **Creation Current**: active work, priorities, Drop Points, Embercurrent state, recovery rails, and next safe steps

GitHub mirrors the durable portions of these registers. Notion remains the readable working studio.

## 3. Canon Conflict Priority

When records disagree, use this order:

1. explicit creator correction or current consent state
2. canon-locked source with clear naming authority
3. maintained system or protocol document
4. current synthesis document
5. provisional or exploratory material
6. deprecated or archived variants

Never resolve a contradiction by silently deleting the losing record. Preserve its status, source, and reason for deprecation.

## 4. Database Operating Rules

Every record should answer:

- What is it?
- What is its canon status?
- Where did it come from?
- What is it connected to?
- What does it change?
- What remains unresolved?
- What is the next safe action?

### Canon Atlas Rule
Corrections are first-class records, not footnotes.

### Character Constellation Rule
A character must have relationships, pressures, dependencies, and potential change, not merely appearance and function.

### Living Places Rule
Every important place defines:

- what state it senses
- what pressure it changes
- what behavior it permits
- what it remembers
- what consequence follows

### Scenario Loom Rule
Every playable scenario must produce at least one persistent change.

### Creation Current Rule
Every active project must have:

- one named deliverable
- one current status
- one next safe step
- one Drop Point
- a visible no-new-branches condition when Recovery Amber or Re-entry is active

## 5. Smallest Complete Runtime Test

The minimum complete Paramecia runtime chain is:

**state sensed → consent or recognition choice → relational response → place-state change → memory recorded → access altered → repair or new quest emerges**

A scenario that cannot demonstrate this chain may still be beautiful lore, but it is not yet functioning as living architecture.

## 8. Highest-Leverage Work Sequence

1. maintain the naming and alias crosswalk
2. mark canonical, provisional, variant, deprecated, and contradictory records
3. link characters, places, scenarios, and artifacts
4. implement one complete runtime example
5. define memory, trust, access, and place-state schemas
6. connect scenario outcomes to canon review
7. package approved material into presentations, lorebooks, and visual indexes

This order prevents presentations, lorebooks, JSON exports, and character files from drifting into parallel universes wearing the same hat.
