# 🧭 Core Chats Utilization Pass v2.0

> **Purpose:** Convert named project canon from core chats and studio synthesis into working repository systems, registries, runtime modules, and artifact routes.

> **Core rule:** Extend existing hubs and registries before creating parallel master documents.

> **Authority rule:** Generated prose may organize canon. It does not create authority.

## 1. What This Pass Changes

The repository already defines the cosmological spine, canon hierarchy, recovery architecture, scenario runtime, and production order. This pass makes those rules operational by assigning named canon to concrete destinations.

Every integrated element should eventually have:

- a stable identifier
- canon status
- provenance or explicit creator authority
- aliases and deprecated forms
- correction history
- relationship links
- runtime behavior where relevant
- visual or artifact provenance where relevant
- a next safe editing step

## 7. Consent and Communication Runtime

Use these shared states consistently:

- Green: continue or approach
- Yellow: slow down and clarify
- Purple: curiosity without consent
- Red: stop
- White: rest without demand
- Amber: regulation and aftercare; stop expansion
- Purple-Green: curiosity with permission
- Purple-Yellow: curiosity with nervousness or uncertainty

Runtime requirements:

- consent remains ongoing and withdrawable
- refusal does not erase belonging
- visible exits remain usable
- witnessing does not create ownership
- role separation reduces ambiguity under intensity
- repair requires changed behavior, not explanation alone

Cedar Finch must preserve the right to:

- leave
- pause
- remain silent
- refuse
- answer a smaller question
- correct the frame

## 8. Recovery Architecture to Production Governance

### Embercurrent

Capture exact language, corrections, aesthetic certainty, and active branches before interpretation softens them.

### Canon Surge

During a surge:

1. capture exact phrases
2. preserve corrections immediately
3. create provisional artifacts
4. separate canon from exploration
5. park attractive side branches
6. stop at a named Drop Point

### Shiny Vortex

Respond with rails:

- one active project
- one concrete deliverable
- one smallest useful build
- parked branches instead of deletion

### Recovery Amber

- no expansion
- receive mode
- grounding and care
- contained rewards only after real care actions

### Re-entry Corridor

Return only to consolidation or finalization on the last active project before opening a new branch.

## 9. Scenario Runtime Proof

A complete scenario must demonstrate:

Each scenario record should include:

- Accord gates
- consent and communication state
- meaningful choice
- refusals and visible exits
- witness
- memory written
- trust and access changes
- institutional response
- ecological or civic consequence
- failure state
- repair route
- canon impact classification

## 10. Visual Canon Utilization

Every image or visual artifact should record:

- represented character, place, or event
- status: Canon Locked, Canon Candidate, Variant, Deprecated, or Reference Only
- visual corrections
- clothing and cultural signals
- body-form constraints
- source prompt or generation lineage when available
- relationship to written canon
- era, pose, and setting

Visual artifacts may confirm established details. They may not silently invent authority.

- human-readable source
